# 2.3.2. Diagrama de Casos de Uso

## Introdução
O Diagrama de Casos de Uso é uma ferramenta da UML essencial para a fase de levantamento e análise de requisitos de um projeto. Ele oferece uma visão de alto nível das funcionalidades do sistema, mostrando o que o software faz a partir da perspectiva de quem o utiliza.

Enquanto outros diagramas UML (como diagramas de classes, sequência ou atividades) detalham a estrutura ou o comportamento interno do sistema, o Diagrama de Casos de Uso começa pela perspectiva externa, mostrando quem interage com o sistema e quais funcionalidades são acessíveis a esses atores. Essa abordagem é particularmente útil para elicitação e comunicação de requisitos funcionais ([Sommerville (2019)](https://www.facom.ufu.br/~william/Disciplinas%202018-2/BSI-GSI030-EngenhariaSoftware/Livro/engenhariaSoftwareSommerville.pdf)).

## Metodologia
A criação do Diagrama de Casos de Uso foi baseada diretamente nos requisitos funcionais e nas histórias de usuário ("User Stories") definidas para o projeto. A metodologia seguiu estas etapas principais:

1. Identificação de Atores: Os atores foram as primeiras entidades a serem identificadas. Eles representam as pessoas ou sistemas externos que interagem com o software. Neste projeto, os atores principais são o Usuário e o Sistema Externo de Calendário.

2. Definição de Casos de Uso: As funcionalidades do sistema (as "User Stories") foram agrupadas em casos de uso de alto nível. Por exemplo, todas as ações de manipulação de tarefas (criar, editar, excluir, finalizar) foram consolidadas no caso de uso Gerenciar Tarefas.

3. Estabelecimento de Relacionamentos: As associações entre os atores e os casos de uso foram definidas para mostrar quem é o responsável por executar cada funcionalidade. As relações de inclusão (*<< include >>*) e extensão (*<< extend >>*) foram usadas para manter a clareza e a simplicidade do diagrama de alto nível.

4. Modelagem da Fronteira do Sistema: Uma caixa foi utilizada para delimitar a fronteira do sistema, com todos os casos de uso posicionados dentro dela. Os atores foram colocados do lado de fora, indicando sua interação externa com o sistema.

Esta metodologia garantiu um diagrama claro e conciso, que serve como um guia para a equipe entender as funcionalidades do sistema e suas responsabilidades.

### Especificação de Casos de Uso
A especificação de casos de uso é um artefato complementar ao diagrama de casos de uso, responsável por detalhar, em formato textual, as funcionalidades representadas graficamente. Enquanto o diagrama apresenta uma visão geral das interações entre atores e o sistema, a especificação descreve de forma narrativa e estruturada como essas interações ocorrem, permitindo maior clareza e precisão na definição dos requisitos funcionais.

Cada especificação normalmente contempla informações como:

- Nome do caso de uso;
- Descrição da funcionalidade;
- Atores envolvidos;
- Pré-condições e pós-condições;
- Fluxo principal de execução;
- Fluxos alternativos e de exceção.

Esse documento é essencial para alinhar a compreensão entre analistas, desenvolvedores, testadores e usuários, servindo como base para implementação e para a elaboração de casos de teste. Além disso, garante que os requisitos funcionais estejam bem descritos e reduz ambiguidades no processo de desenvolvimento de software.

Segundo [Sommerville (2019)](https://www.facom.ufu.br/~william/Disciplinas%202018-2/BSI-GSI030-EngenhariaSoftware/Livro/engenhariaSoftwareSommerville.pdf) e [Pressman (2016)](https://archive.org/details/pressman-engenharia-de-software-uma-abordagem-profissional-8a), a utilização de casos de uso e suas especificações contribui para uma comunicação mais eficaz entre equipe técnica e não técnica, uma vez que descreve o comportamento esperado do sistema em linguagem acessível, mas suficientemente detalhada. A UML, conforme definida pelo Object Management Group (OMG, 2015), formaliza essa prática e estabelece sua importância no ciclo de engenharia de requisitos.
